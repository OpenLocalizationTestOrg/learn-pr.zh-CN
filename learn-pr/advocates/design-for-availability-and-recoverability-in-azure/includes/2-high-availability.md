高可用性 (HA) 可确保体系结构能够处理故障。 假设您负责必须始终能够完全操作的系统。 可能会发生故障并将发生故障, 因此如何确保系统在出现问题时保持在线状态？ 如何在不中断服务的情况下执行维护？ 

在这里, 你将了解高可用性的需求, 评估应用程序高可用性要求, 并了解 Azure 平台如何帮助你实现可用性目标。

## <a name="what-is-high-availability"></a>高可用性是什么？

> [!VIDEO https://www.microsoft.com/videoplayer/embed/RE2yEvc]

高可用性服务是一项 absorbs 相关服务和硬件中的可用性、负载和临时故障的变化的服务。 在执行 acceptably 时, 应用程序仍保持联机并可用 (或保持其外观)。 此可用性通常由业务要求、服务级别目标或服务级别协议来定义。

高可用性最终是指处理系统组件的丢失或严重降级的能力。 这可能是因为主机发生故障, 托管应用程序将脱机的虚拟机。 这可能是由于系统升级计划的维护造成的。 它甚至可能是由云中的服务故障导致的。 确定系统可能发生故障的地方, 并构建用于处理这些故障的功能, 将确保您为客户提供的服务可以保持在线。

服务的高可用性通常需要构成服务的组件的高可用性。 考虑提供在线市场以购买项目的网站。 向客户提供的服务是能够在联机状态下列出、购买和销售项目的功能。 若要提供此服务, 您将拥有多个组件: 数据库、web 服务器、应用程序服务器等。 这些组件中的每一个都可能会失败, 因此您必须确定故障点的方式和位置, 并确定如何在体系结构中解决这些故障点。

## <a name="evaluate-high-availability-for-your-architecture"></a>评估体系结构的高可用性

评估应用程序的高可用性的步骤有三个:

1. 确定应用程序的服务级别协议
1. 评估应用程序的 HA 功能
1. 评估依赖应用程序的 HA 功能

让我们详细了解这些步骤。

### <a name="determine-the-service-level-agreement-of-your-application"></a>确定应用程序的服务级别协议

服务级别协议 (SLA) 是服务提供商和服务使用者之间的一种协议, 服务提供商根据可衡量的指标和定义的责任向服务提供商提交标准服务。 sla 可以是严格、法律约束、合同协议或预期的客户的预期可用性。 服务指标通常重点关注服务吞吐量、容量和可用性, 所有这些都可以通过多种方式衡量。 无论构成 sla 的具体指标如何, 如果无法满足 sla, 都可能会对服务提供商产生严重的财务后果。 服务协议的一个公用组件保证了财务补偿, 缺少 sla。

服务级别目标 (SLO) 是用于衡量性能、可靠性或可用性的目标指标的值。 这些指标可以是定义请求处理性能的指标 (以毫秒为单位)、服务的可用性 (每月分钟数) 或每小时处理的请求数。 通过评估您的应用程序公开的指标并了解客户用作质量指标的指标, 您可以为这些 slo 定义可接受和不可接受的范围。 通过定义这些目标, 可以清楚地为支持使用这些服务的服务和客户的团队设置目标和期望。 这些 slo 将用于确定是否符合你的总体 SLA。

下表显示了各种 SLA 级别可能造成的累积故障时间。 

| SLA | 每周停机时间 | 每月停机时间 | 每年停机时间 |
| --- | --- | --- | --- |
| 99% |1.68 小时 |7.2 小时 |3.65 天 |
| 99.9% |10.1 分钟 |43.2 分钟 |8.76 小时 |
| 99.95% |5分钟 |21.6 分钟 |4.38 小时 |
| 99.99% |1.01 分钟 |4.32 分钟 |52.56 分钟 |
| 99.999% |6秒 |25.9 秒 |5.26 分钟 |

当然, 可用性越高越好, 其他一切都是相等的。 但是, 随着您在努力实现更多的 9, 成本和复杂性可实现该级别的可用性增长。 99.99% 的正常运行时间约为每月总停机时间的5分钟。 是否值得将额外的复杂性和成本推向五个9？ 答案取决于业务要求。 

以下是定义 SLA 时的一些其他注意事项:

* 若要实现四个 9 (99.99%), 您可能无法依靠手动干预从故障中恢复。 应用程序必须是自诊断和自我修复。 
* 除了四个 9, 更难快速检测出故障以满足 SLA 要求。
* 考虑评估 SLA 所依据的时间范围。 窗口越小, 公差越紧密。 在每小时或每日正常运行时间定义 SLA 可能不是很有意义的。 

确定您的体系结构将需要的高可用性功能时, 确定 sla 是重要的首要步骤。 这些方法将帮助您将用来使应用程序高度可用的方法进行整形。

### <a name="evaluate-the-ha-capabilities-of-the-application"></a>评估应用程序的 HA 功能

若要评估应用程序的 HA 功能, 请执行故障分析。 重点关注单个故障点和关键组件, 这些组件对应用程序的访问、配置不当或启动行为意外导致的影响较大。 对于具有冗余的区域, 确定应用程序是否能够检测错误条件和自我修复。

您需要仔细评估应用程序的所有组件, 包括旨在提供 HA 功能的部分, 如负载平衡器。 需要修改单一故障点, 以使 HA 功能集成, 或者需要将其替换为可提供 ha 功能的服务。

### <a name="evaluate-the-ha-capabilities-of-dependent-applications"></a>评估依赖应用程序的 HA 功能

您不仅需要了解您的应用程序对您的使用者的 sla 要求, 还需要了解应用程序可能依赖的任何资源的提供 sla。 如果要向客户提交 99.9% 的正常运行时间, 但您的应用程序所依赖的服务的运行时间承诺仅为 99%, 这可能导致您无法满足您的客户 SLA 的风险。 如果依赖服务无法提供足够的 sla, 您可能需要修改您自己的 sla, 将依赖项替换为替代项, 或者在依赖项不可用时查找符合 SLA 的方法。 根据方案和依赖项的性质, 可以暂时使用类似于缓存和工作队列的解决方案来解决依赖项的故障。

## <a name="azures-highly-available-platform"></a>Azure 的高可用性平台

Azure 云平台设计为可在其所有服务中提供高可用性。 与任何系统一样, 应用程序可能受硬件和软件平台事件的影响。 设计应用程序体系结构以处理故障的需求非常关键, Azure 云平台为您提供了使应用程序高度可用的工具和功能。 在为 Azure 上的体系结构考虑 HA 时, 有以下几个核心概念:

* 可用性集
* 可用性区域
* 负载平衡
* 平台即服务 (PaaS) HA 功能

### <a name="availability-sets"></a>可用性集

可用性集是一种通知 Azure 的方法。应分发属于相同应用程序工作负荷的 vm, 以防止硬件故障和定期维护的并发影响。 可用性集由*更新域*和*容错域*组成。

![显示三个可用性集的插图。 第一个集有一个更新域, 第二个更新域有两个更新域, 第三个是没有任何更新域的。](../media/AzAvailSets.png)

更新域: 确保当虚拟机托管在 Azure 数据中心中时, 您的应用程序服务器的子集始终保持运行状态, 以便在维护时需要停机。 大多数更新可以在不影响其上运行的 vm 的情况下执行, 但有时这是不可能的。 为了确保一次更新不会对整个数据中心进行更新, Azure 数据中心在逻辑上分为多个更新域 (UD)。 当需要将维护事件 (如性能更新和关键安全修补程序) 应用到主机时, 将通过更新域对更新进行排序。 使用更新域进行先后顺序更新可确保在平台更新和修补过程中整个数据中心不可用。

[! 说明] 当 update 域表示数据中心的逻辑部分时, 故障域 (FD) 表示数据中心的物理部分, 并确保在可用性集中的服务器的服务器多样性。 故障域与数据中心中共享硬件的物理分离相一致。 这包括支持位于服务器机架中的物理服务器的电源、冷却和网络硬件。 如果支持服务器机架的硬件变得不可用, 则中断仅影响服务器机架。 通过将 vm 放在可用性集内, 你的 vm 将自动分布在多个 FDs 中, 以便在发生硬件故障时仅有部分 vm 将受到影响。

使用可用性集, 可以确保在需要高影响维护事件或发生硬件故障时, 您的应用程序保持在线。

### <a name="availability-zones"></a>可用性区域

可用性区域是区域内独立的物理数据中心位置, 包括其自己的电源、冷却和网络。 通过在部署资源时考虑可用性区域, 可以保护工作负载不受数据中心中断, 同时在特定区域保留状态。 像虚拟机这样的服务是一种*区域性服务*, 允许您将它们部署到区域内的特定区域。 其他服务是*区域冗余服务*, 将在特定 Azure 区域中的可用性区域之间进行复制。 这两种类型都可确保在 Azure 区域中没有单点故障。

![显示 Azure 区域中三个可用性区域的插图。 每个可用性区域都有自己的一组资源, 并相互连接以进行区域冗余服务的复制。](../media/AzAvailZones.png)

支持的区域至少包含三个可用性区域。 在这些区域中创建区域性服务资源时, 您将能够选择应在其中创建资源的区域。 这将使您能够将应用程序设计为经受区域性中断, 并继续在 azure 区域中运行, 然后再将您的应用程序 evacuate 到另一个 azure 区域。

可用性区域是 Azure 区域的较新的高可用性配置服务, 目前可用于某些区域。 如果您想要考虑此功能, 请务必在计划部署应用程序的区域中检查此服务的可用性。 使用虚拟机以及多个 PaaS 服务时支持可用性区域。 可用性区域与可用性集互斥。 使用可用性区域时, 不再需要为系统定义可用性集。 你将在数据中心级别拥有多元化, 并且将永远不会对多个可用性区域同时执行更新。

### <a name="load-balancing"></a>负载平衡

负载平衡器管理如何在应用程序中分布网络流量。 负载平衡器对于保持应用程序对单个组件故障的恢复能力非常重要, 并确保应用程序可用于处理请求。 对于未内置服务发现的应用程序, 可用性集和可用性区域都需要负载平衡。

Azure 拥有三种负载平衡技术服务, 它们在功能上不同于路由网络流量:

* **Azure 流量管理器**提供全局 DNS 负载平衡。 您可以考虑使用流量管理器在 Azure 区域内或跨 Azure 区域提供 DNS 终结点的负载平衡。 流量管理器会将请求分发到可用终结点, 并使用终结点监视来检测和删除加载失败的终结点。
* **Azure 应用程序网关**提供了第7层负载平衡功能, 如传入流量的循环分布、基于 cookie 的会话相关性、基于 URL 路径的路由以及在单个应用程序中托管多个网站的功能应用程序网关。 [! 注意] 默认情况下, 应用程序网关监视其后端池中所有资源的运行状况, 并自动从池中删除任何被视为不正常的资源。 应用程序网关继续监视不正常的实例, 并在这些实例变为可用后再将它们重新添加到正常的后端池, 并对运行状况探测器做出响应。
* **Azure 负载平衡器**是第4层负载平衡器。 您可以配置公共和内部负载平衡终结点, 并定义规则, 以使用 TCP 和 HTTP 运行状况探测选项来映射到后端池目标的入站连接, 以管理服务可用性。

这三种 Azure 负载平衡技术的一个或多个组合可确保您有必要的选项可用于构建高可用性解决方案, 以通过应用程序路由网络流量。

![azure 负载平衡 optionsAn 图显示了 Azure 中的不同负载平衡技术。 流量管理器平衡两个区域之间的负载。 在每个区域中, 都有一个应用程序网关, 它根据请求类型在 web 层中分布不同虚拟机之间的负载。 所有图像请求都将转到映像服务器池, 任何其他请求将定向到默认服务器池。 来自默认服务器池的进一步请求由 Azure 负载平衡器处理, 以在数据库层中的虚拟机之间分发它们。](../media/AzLBOptions.png)

### <a name="paas-ha-capabilities"></a>PaaS HA 功能

提供的 PaaS 服务具有内置的高可用性。 azure SQL Database、azure App Service 和 azure Service Bus 等服务包括高可用性功能, 并确保服务的单个组件的故障对您的应用程序是无缝的。 使用 PaaS 服务是确保体系结构高度可用的最佳方法之一。

在构建高可用性时, 您需要了解提交给客户的 SLA。 然后, 评估您的应用程序具有的 ha 功能以及依赖系统的 ha 功能和 sla。 建立这些功能之后, 请使用 Azure 功能 (如可用性集、可用性区域和各种负载平衡技术) 将 HA 功能添加到应用程序中。 应选择使用的任何 PaaS 服务都具有内置的 HA 功能。
